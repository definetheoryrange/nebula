# ResourceForge

ResourceForge 是一个面向开发者、技术研究人员与内容策展人的轻量化外链资源汇总与导航工具。项目定位为“技术化书签管理器”，用于解决个人或团队在多个项目、多个知识领域间分散收藏、难以检索、缺乏结构化组织方式的资源管理问题。ResourceForge 不提供爬虫、采集或自动抓取功能，而是以人工策展与标签化组织为核心，强调资源条目的可维护性、可追溯性与可分享性。

ResourceForge 适用于需要高频查阅外部技术文档、API 参考、社区讨论、行业动态或特定主题资源集合的开发人员，也适用于技术团队内部建立统一的公共知识库入口。项目本身不依赖数据库或重型后端服务，基于静态 Markdown 与轻量级元数据文件运作，可部署在任何支持静态托管的平台上，同时保留通过 CLI 工具进行索引生成与校验的能力。

## 功能概览

- **资源条目结构化存储**：每条资源支持标题、原始 URL、标签、归属分类、添加日期与备注说明，以 YAML Front Matter 或独立元数据文件形式保存，便于版本管理与批量编辑。

- **多级分类与标签过滤**：支持为资源分配多个标签，并建立两级分类体系（领域/子领域）。导航页面可根据分类或标签动态生成索引视图，支持按名称、日期、标签组合进行过滤。

- **外链健康检查**：内置基于 HTTP HEAD 请求的链接可用性检测工具，可定期扫描资源列表中的全部外链，识别返回 4xx/5xx 状态码或超时的条目，并生成检测报告。

- **导航页面静态生成**：提供命令行生成器，根据资源元数据自动生成静态 HTML 导航页面，支持响应式布局与明暗主题，无需每次手动更新页面代码。

- **资源变更审计日志**：记录每条资源的添加、修改、删除操作，包含操作时间与操作者标识，便于团队协作场景下的变更追溯与误操作回滚。

- **导入与导出兼容性**：支持从浏览器书签 HTML 导出文件、Markdown 列表文件、CSV 表格文件导入资源条目，也支持将当前资源库导出为上述格式，便于迁移或备份。

- **自定义元数据扩展**：允许用户为资源定义自定义字段（如阅读状态、重要程度、关联项目编号），并在索引视图中展示或过滤这些扩展字段。

## 应用场景

- **技术团队内部知识库入口管理**：开发团队可将项目相关的 API 文档、设计规范、运维手册、日志查询平台等外链统一收录至 ResourceForge，每位成员通过导航页面快速访问，新成员入职时可一次性查阅全部必需外部资源。

- **开源项目维护者的外部依赖追踪**：开源项目维护者可使用 ResourceForge 记录项目依赖的第三方库文档、社区论坛、问题追踪器、CI/CD 服务控制台等链接，并定期运行健康检查，及时发现已失效或迁移的依赖地址。

- **技术博主或内容策展人的话题资源集**：技术博主在撰写系列文章或准备技术分享时，可将调研过程中收集的参考文献、示例项目、数据来源、在线工具等统一整理为资源集，通过 ResourceForge 生成可公开分享的导航页面。

- **个人学习路径的阶段性资源归档**：学习新技术时，可将在线课程入口、官方教程、练习平台、社区讨论精华帖、代码示例仓库等按学习阶段分类保存，后续复习或整理笔记时可快速定位。

## 快速开始

执行以下命令从 GitHub 克隆项目、安装依赖并启动本地开发服务：

```bash
git clone https://github.com/resourceforge/resourceforge.git
cd resourceforge
npm install
npm run build
npm run serve
```

执行后，静态导航页面默认生成于 `dist/` 目录，本地开发服务默认监听 `http://localhost:8080`。如需自定义资源数据路径或输出目录，可编辑项目根目录下的 `config.yaml` 文件。

## 贡献指南

1. 在 GitHub 上 Fork 本仓库，并克隆到本地开发环境。新建分支时请使用 `feature/` 或 `fix/` 前缀，并简要描述变更内容。

2. 本地开发时，请确保通过 `npm run test` 运行全部单元测试与集成测试，新增功能需附带对应的测试用例。提交前执行 `npm run lint` 进行代码风格检查。

3. 若您希望新增导入或导出格式支持，请在 `src/importers/` 或 `src/exporters/` 下新增模块，并参照现有模块实现 `parse()` 或 `generate()` 方法，同时更新 `docs/developer-guide/` 中的对应章节。

4. 提交 Pull Request 时请填写清晰的变更描述，说明解决的问题、实现方式以及是否影响现有配置或数据格式。若变更涉及用户可见行为，请在 PR 描述中标注是否需要更新文档。

5. 如发现资源链接失效或示例数据存在问题，请先通过 Issue 报告，确认问题后再提交修复代码。

## 常见问题

**问：ResourceForge 是否支持多用户同时编辑同一资源库？**

答：ResourceForge 本身不提供内置的并发锁机制或在线协作界面。若需多人协作，建议将资源数据目录置于 Git 版本控制下，通过分支策略和 Merge Request 流程协调变更。对于频繁冲突的元数据文件，可考虑按资源条目拆分文件（每条资源独立 .yaml 文件）以降低冲突概率。

**问：健康检查功能会频繁请求外链，是否会被目标网站封禁？**

答：健康检查默认采用串行请求，并设置 5 秒超时与 2 秒请求间隔。您可在配置文件中调整 `healthCheck.concurrentLimit` 与 `healthCheck.intervalMs` 参数。建议将检查频率设置为每周一次，且避免在目标站点业务高峰时段运行。若检测到目标站点返回 429 状态码，工具会自动暂停当前批次并延长间隔。

**问：导航页面生成后，资源变更是否需要重新生成全部页面？**

答：增量生成功能目前处于实验阶段，默认每次执行 `npm run build` 会重新生成全部静态页面。对于资源数量超过 5000 条的大型资源库，建议在配置中启用 `build.incremental` 选项，生成器将基于资源文件的最后修改时间戳判断是否需要重新生成对应索引页面。
