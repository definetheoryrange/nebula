# OpenResource Hub

OpenResource Hub 是一个面向技术开发者与开源爱好者的外链资源聚合与导航系统。本项目并非传统的内容管理系统，而是一套以技术资源目录为核心、以外部优质信息源为载体的轻量级导航框架。目标用户包括个人技术博客运营者、开源项目维护者、技术社区编辑以及希望高效追踪特定领域信息源的研究人员。项目解决的核心问题是：在信息过载的互联网环境中，如何通过结构化链接管理与分类索引，快速定位到高价值的外部技术资源，并降低重复检索与人工筛选的时间成本。

系统采用纯静态架构，所有资源条目以标记化格式存储，支持按领域、地区、赛事类型、数据频率等多维度筛选。同时提供可插拔的校验钩子，用于检测外部链接的可达性与状态码变化，帮助维护者及时发现失效或迁移的资源。项目本身不存储任何第三方内容，仅保留指向原始信息源的引用元数据，符合开源合规要求。

## 功能概览

- **多级分类索引**：支持按地理区域、赛事等级、数据类型创建无限层级的分类树，每个分类可挂载多个外部链接，并自动生成面包屑导航。
- **链接状态监控**：内置轻量级 HTTP 状态检查工具，可定时探测每个收录链接的响应码、响应时间与重定向链，异常时输出告警日志。
- **元数据标签系统**：每个资源条目可附加自定义键值对元数据（如语言、更新频率、是否需要代理），便于实现高级筛选与搜索。
- **批量导入导出**：支持从 CSV 或 JSON 文件批量导入链接清单，也可将当前索引导出为标准化格式，用于备份或迁移。
- **静态站点生成**：基于模板引擎将资源索引渲染为纯静态 HTML 页面，可直接部署至对象存储或 CDN，无需数据库与后端服务。
- **访问统计看板**：通过解析 Web 服务器访问日志，生成各资源链接的点击热度与来源分布，辅助维护者判断资源价值。
- **链接关系图谱**：自动分析不同资源之间的相互引用关系，生成简单的依赖网络图（基于引用头或显式关联配置）。
- **自定义校验脚本**：允许用户编写 JavaScript 或 Shell 钩子，在链接更新或新增时执行自定义验证逻辑（如检查页面关键内容是否存在）。

## 应用场景

1. **体育赛事技术数据聚合**：项目维护者可以收录各类洲际或地区性体育赛事的实时比分、积分榜与赛程接口，形成统一的技术数据入口，供数据分析团队或预测模型使用。
2. **开源社区文档导航**：开源项目文档站可使用本系统整理外部参考链接，例如相关规范文档、参考实现仓库、社区讨论区等，帮助新贡献者快速了解生态全貌。
3. **学术研究资源管理**：研究人员可将不同地区、不同机构的公开数据集或统计报告链接集中管理，并通过状态监控功能及时获知数据源是否下线或变更。
4. **运维监控告警联动**：运维团队可将本系统作为内部知识库的一部分，收录各类状态页、监控面板与云服务健康状态链接，配合检查脚本实现自动告警。
5. **本地化内容镜像指引**：面向跨国业务时，可收录不同区域专属的域名或服务入口，通过分类标签明确适用范围，避免团队成员访问错误的地区端点。

## 快速开始

以下步骤适用于 Linux 或 macOS 环境，Windows 用户建议使用 WSL 或 Git Bash。

```bash
# 1. 克隆代码仓库
git clone https://github.com/your-org/openresource-hub.git
cd openresource-hub

# 2. 安装依赖（项目使用 Node.js 18+ 与 pnpm）
npm install -g pnpm
pnpm install

# 3. 复制环境配置模板并填充资源目录
cp .env.example .env
pnpm run init:index

# 4. 启动开发服务器（默认监听 3000 端口）
pnpm run dev
```

启动成功后，访问 `http://localhost:3000` 即可查看本地的资源导航界面。如需生成静态站点，执行 `pnpm run build`，产物位于 `dist/` 目录。

## 贡献指南

我们欢迎各类形式的贡献，包括但不限于新增资源链接、改进解析器性能、完善文档以及修复缺陷。请遵循以下流程：

1. 在 GitHub 上 fork 本仓库，并克隆到本地开发环境。确保使用 `main` 分支的最新代码作为基准。
2. 创建一个具有描述性名称的功能分支，例如 `feature/add-south-asia-category` 或 `fix/csv-parser-encoding`。
3. 若新增或修改资源链接，请同时更新 `data/sources.csv` 文件，并在 `data/metadata/` 中补充对应的元数据 JSON，确保字段完整。
4. 提交前运行 `pnpm run test` 确保所有单元测试通过，并执行 `pnpm run lint` 检查代码风格。新增功能需附带相应的测试用例。
5. 提交 Pull Request 时，请清晰描述变更目的、测试覆盖情况及是否影响现有功能。PR 描述中请关联相关 Issue（若有）。

## 常见问题

**问：如何批量更新所有链接的状态？**

答：项目提供了命令行工具，执行 `pnpm run check:all` 即可遍历 `sources.csv` 中的所有链接，并发执行 HTTP 头部请求，输出结果到 `logs/check-report.json`。您也可以配置 cron 任务定期执行该命令，并结合告警脚本通知异常。

**问：我收录的链接需要登录或携带 token 才能访问，如何处理？**

答：系统支持在元数据中配置 `headers` 字段，您可以在 `data/metadata/{id}.json` 中添加自定义请求头，例如 `{"Authorization": "Bearer xxxx"}`。校验器会自动合并这些头信息。请注意保管敏感信息，建议使用环境变量替换实际值。

**问：生成的静态站点是否可以部署到 GitHub Pages？**

答：完全可以。执行 `pnpm run build` 后，将 `dist/` 目录下的所有文件推送至仓库的 `gh-pages` 分支，并在仓库设置中启用 Pages 功能即可。默认的构建产物包含相对路径资源，无需额外配置即可正常显示。
