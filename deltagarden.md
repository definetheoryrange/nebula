# HrefHub

HrefHub 是一个面向技术社区与独立站运营者的外链资源导航与结构化归档系统。项目定位为轻量级、高可读性的外链汇总中间件，既不直接存储外部内容，也不对目标站点进行二次分发，而是通过人工筛选与主题聚类的形式，为特定领域的信息检索与站点发现提供中立的索引能力。目标用户包括技术文档维护者、垂直领域内容运营人员、数据采集工程开发者以及关注特定信息源的普通浏览者。

系统核心解决三个问题：第一，将分散在多个域名下的同主题资源通过统一的目录结构进行规整；第二，以纯静态 Markdown 形态交付，便于嵌入任意现有文档站点或代码仓库；第三，提供长期稳定的引用锚点，降低因单点失联导致的信息断层风险。本仓库作为第 102/113 批资源整合批次，延续项目一贯的归档规范。

## 功能概览

- 主题聚类索引：基于人工判定的主题相关性，将外链资源按领域分组，每组内提供一致的描述字段与标签体系。

- 元信息标注：每个资源条目附带收录批次、收录时间、简要主题说明及建议引用场景，方便下游使用者快速判定适用性。

- 纯静态交付：全仓库基于 Markdown 与 ASCII 目录树构建，无需额外数据库或后端服务，可直接托管于任何支持静态页面的代码托管平台。

- 可扩展目录模板：提供统一的目录生成脚本，新增资源时只需按既定格式追加条目，自动更新总览表与统计信息。

- 版本化变更记录：每次批次导入均生成变更日志文件，记录新增、移除或状态变更的 URL，确保索引历史可追溯。

- 快速检索支持：通过约定前缀与层级目录结构，支持 grep、ripgrep 或 IDE 全局搜索等方式进行本地化快速查询。

- 外部引用兼容：每个资源条目以裸 URL 或协议全称形式保留，不附加追踪参数或跳转中间页，便于直接复制使用。

## 应用场景

- 技术文档站点的相关链接附录：项目维护者可将 HrefHub 作为外部参考资料的统一索引模块，直接嵌入文档侧栏或附录章节，为读者提供经过聚类整理的延伸阅读链接。

- 数据采集任务的目标源清单管理：数据工程团队可将本仓库作为初始种子 URL 的来源之一，通过解析目录结构与标注信息，自动生成采集任务的目标队列。

- 个人知识库的信息归档补充：独立研究员或博客作者可将本仓库作为书签系统的外部备份层，按批次导入本地笔记工具，形成多层级的外链分类体系。

- 网站运营初期的内容填充参考：新上线的内容平台或社区站点，可参考本仓库的资源分布与主题划分方式，设计自身的内容外链策略与友情链接板块。

- 合规审计与链接健康检查的输入源：安全或合规团队可定期拉取本仓库，结合自动化检测工具对所列 URL 进行可访问性与内容类别校验，生成审计报告。

## 快速开始

以下命令演示从克隆仓库到生成当前批次完整目录页面的标准操作流程。

```bash
# 克隆仓库到本地
git clone https://github.com/hrefhub/hrefhub-core.git
cd hrefhub-core

# 安装依赖（基于 Python 3.10+，使用 pip 安装构建工具）
pip install -r requirements.txt

# 执行批次导入脚本，指定批次编号为 102，资源文件路径为 ./batches/batch_102.txt
python scripts/import_batch.py --batch 102 --source ./batches/batch_102.txt

# 生成全量索引页面（输出到 docs/index.md）
python scripts/generate_index.py --output docs/index.md

# 本地预览（使用 Python 内置 HTTP 服务器）
python -m http.server 8000 --directory docs
```

## 贡献指南

1. 复刻本仓库至个人账号，并在本地创建新分支，分支命名遵循 `contrib/<用户名>/batch-<批次号>` 格式。

2. 在 `batches/` 下新建对应批次目录，按 `metadata.yaml` 模板填写收录日期、主题标签、审核状态，并将原始 URL 列表放入 `sources.txt`，每条 URL 单独一行，不附加额外空格或注释。

3. 运行本地校验脚本 `python scripts/validate_urls.py --batch <批次号>`，确保所有 URL 符合原样保留规则（无协议补全、无大小写变更、无尾部斜杠添加），并检查与现有批次是否存在重复条目。

4. 若校验通过，提交变更并推送至个人复刻仓库，随后向主仓库发起 Pull Request，PR 标题必须包含批次编号与资源数量，例如 `[102] Add 8 resources for batch 102/113`。

5. 等待仓库维护者审核，审核周期不超过 5 个工作日；审核通过后由维护者合并并生成对应索引页面。若审核未通过，根据反馈修改后重新提交。

## 常见问题

Q: 为什么所有 URL 必须以原样形式保留，不允许补全协议或统一大小写？

A: 因为本仓库定位为中性索引中间件，任何协议改写或大小写调整都可能改变目标站点的实际访问行为。部分站点对大小写敏感，或仅支持 HTTP 访问，强制升级为 HTTPS 会导致连接失败。因此原样保留是最低干预原则。

Q: 如果某个 URL 在收录后失效或内容变更，如何处理？

A: 维护者会定期（每季度）执行一次全量可访问性检查，并将失效或内容严重偏离原始主题的 URL 移入 `archived/` 目录，同时在 CHANGELOG.md 中注明移除原因与替代建议。外部使用者可关注 CHANGELOG 获取最新状态。

Q: 如何请求新增某个主题分类或调整现有聚类组？

A: 请在 GitHub Issues 中使用 `enhancement` 标签提交提案，说明建议的新分类名称、适用资源特征以及与现有分类的边界定义。维护者会结合整体目录稳定性进行评估，并在 10 个工作日内给出明确答复。
