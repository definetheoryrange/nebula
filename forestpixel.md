# ResourceHub

ResourceHub 是一个面向开发者与技术研究人员的开源外链资源聚合与导航系统。项目定位为高质量技术站点的结构化收录与快速检索工具，通过人工筛选与社区贡献相结合的方式，维护一份可机读、可扩展的 URL 资源清单，帮助用户摆脱低效的搜索引擎筛选过程，直接触达垂直领域内的有效信息源。本项目不存储、不托管、不代理任何第三方内容，仅提供链接索引与分类元数据，适用于个人知识管理、团队协作调研、自动化采集任务的前置发现等场景。

## 功能概览

- **多维度资源分类**：按领域、语种、内容形式、更新频率等标签体系对每条 URL 进行标记，支持自定义扩展。
- **批量链接健康检查**：内置周期性可用性探测，自动标记失效链接并生成报告，保证资源池的鲜活度。
- **结构化元数据导出**：支持 JSON、YAML、CSV 格式导出完整资源列表，便于集成到其他自动化流程。
- **标签与全文检索**：基于标题、描述、标签、域名关键词的轻量级检索引擎，响应时间低于 200 毫秒。
- **社区贡献工作流**：提供标准化的新增/修改/删除提案模板，通过 Pull Request 合并外部贡献，变更记录全程可追溯。
- **资源版本快照**：每次更新自动生成快照，支持回滚至任意历史版本，保障生产环境使用的资源列表稳定性。
- **访问统计与热度排序**：记录各链接的点击次数与外部引用情况，辅助判断资源价值与时效性。

## 应用场景

- **技术团队内部知识库建设**：团队可将 ResourceHub 作为基础设施，统一维护项目依赖的文档站、API 参考、镜像源、工具官网等外部链接，新成员入职时可一键获取所有必需的外部资源入口。
- **自动化数据采集任务的前置发现**：数据工程师或爬虫开发者可利用本项目的分类资源作为种子 URL 集合，快速构建垂直领域的站点清单，显著降低初始调研与规则编写的时间成本。
- **个人开发者的技术信息管理**：独立开发者可将日常积累的教程、博客、论坛、开源项目仓库等链接通过本项目的格式进行系统化管理，避免浏览器书签的杂乱与丢失风险。
- **开源社区文档协同维护**：开源项目维护者可借鉴本项目的资源收录规范与贡献流程，为自身项目建立配套的外部链接索引文档，方便社区成员共同维护版本更新。

## 快速开始

以下步骤适用于 Linux / macOS / Windows WSL 环境，确保系统已安装 Git 与 Node.js 18.x 或以上版本。

```bash
# 1. 克隆仓库
git clone https://github.com/resource-hub/resource-hub.git
cd resource-hub

# 2. 安装依赖
npm install

# 3. 初始化本地资源数据库（首次运行）
npm run init-db

# 4. 启动开发服务器
npm run dev

# 5. 执行全量链接健康检查（可选）
npm run check:all
```

访问控制台输出的本地地址（默认 http://localhost:5173）即可浏览资源列表。生产环境部署请参考 `docs/deployment.md`。

## 贡献指南

我们欢迎任何形式的贡献，包括但不限于新增优质资源链接、修正失效条目、改进文档、提交缺陷修复。请遵循以下标准化流程以确保合并效率：

1. **查阅现有资源与标签体系**：在提交新链接前，先通过搜索或浏览分类确认该资源尚未被收录。若已存在，请考虑更新其标签或描述而非重复添加。
2. ** Fork 仓库并创建特性分支**：从主仓库的 `main` 分支 Fork 到个人账户，然后在本地的 `feat/your-feature-name` 分支上进行修改。分支命名建议采用 `feat/`、`fix/`、`docs/` 前缀。
3. **按照模板修改资源文件**：新增或修改资源时，请严格遵循 `data/resources.json` 中的 JSON Schema 结构。每条记录必须包含 `url`、`title`、`category`、`tags`、`description`、`submitter` 字段，并确保 `url` 字段与用户原始输入完全一致（包括协议与大小写）。
4. **运行本地校验与测试**：提交前执行 `npm run validate` 检查数据格式，执行 `npm run test` 确保单元测试通过。若新增外部依赖，请更新 `package.json` 并提交锁文件。
5. **发起 Pull Request**：向主仓库的 `main` 分支发起 PR，并在描述中明确说明本次变更的动机、涉及资源列表的变更范围以及是否经过本地健康检查。PR 通过 CI 校验后，将由维护者进行评审与合并。

## 常见问题

**Q：为什么资源列表中包含的某些域名无法访问？**
A：ResourceHub 仅做链接索引，不保证外部站点的可用性。我们提供每日自动健康检查并在界面中标记异常链接，但最终可用性由目标站点的服务状态决定。若您发现某链接长期失效，欢迎提交移除或替换提案。

**Q：我能否在 ResourceHub 中提交付费或需要注册才能访问的资源？**
A：可以，但必须在资源条目的 `description` 和 `tags` 字段中明确标注「付费」「会员」「注册」等限制条件，以帮助其他用户合理评估。我们建议优先收录公开且稳定的资源，以最大限度提升索引价值。

**Q：项目如何处理资源列表的版本更新与冲突？**
A：所有变更通过 PR 合并进入 `main` 分支，每次合并触发 CI 流水线自动生成带时间戳的 JSON 快照并存储于 `data/snapshots/` 目录。若出现冲突，贡献者需在本地基于最新 `main` 分支 rebase 后解决冲突再提交。生产环境部署时始终引用最新的稳定快照。
