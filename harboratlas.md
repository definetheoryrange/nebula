# NovaLink 技术资源导航站

NovaLink 是一个面向开发人员、数据分析师及运维工程师的轻量级技术资源聚合与导航系统。该项目定位于解决技术从业者在日常工作中检索高质量外部资源效率低、信息分散、链接失效等痛点，通过人工筛选与自动化校验相结合的方式，构建一个高可用、低延迟、分类清晰的外部技术参考库。

NovaLink 并非搜索引擎，亦非内容生成平台，而是一个经过结构化组织的参考信息索引。它帮助用户快速定位到特定领域（如实时数据看板、赛事信息聚合、移动端适配方案等）的可信外部节点，从而缩短信息获取路径，提升研发与决策效率。本项目适用于需要频繁查阅特定类型外部数据的技术团队、个人研究者以及开源项目维护者。

## 功能概览

- **智能分类索引**：按应用领域与技术栈对超过 200 个外部资源节点进行多级标签管理，支持模糊搜索与精确过滤。
- **链接可用性监控**：内置定时校验任务，自动检测外部链接的响应状态码与响应时间，对异常节点进行告警标记。
- **自定义导航面板**：用户可根据自身技术偏好创建独立的资源分组，支持拖拽排序与快捷访问。
- **移动端响应式适配**：前端界面基于 Flexbox 与 Grid 布局，在手机、平板及桌面设备上均提供一致的浏览体验。
- **开放数据导出**：支持将导航列表以 JSON 或 Markdown 表格格式导出，便于嵌入其他文档或自动化工具。
- **多维度排序与筛选**：支持按域名、可用性、更新时间、访问热度等维度对资源进行动态排序。
- **历史快照回溯**：对关键外部资源节点保留近 30 天的可用性历史记录，便于排查故障时间线。
- **轻量级部署**：后端基于 Node.js + SQLite，无需外部数据库依赖，单机即可运行，资源占用低于 128 MB 内存。

## 应用场景

- **技术团队内部知识库增强**：技术负责人可将 NovaLink 部署为团队内部的统一入口，集中管理项目常用的第三方 API 文档、监控面板与数据源地址，避免成员各自收藏导致信息孤岛。
- **数据聚合类项目的辅助导航**：从事体育数据、实时比分或竞猜类应用的开发者，可利用本导航站快速访问多个外部数据发布节点，作为自身数据源的备用参考通道。
- **开源文档站的外链补充**：开源项目维护者可在项目的 README 或 Wiki 中引用 NovaLink 的导出功能，为用户提供经过筛选的扩展阅读或数据参考链接，提升文档的实用性。
- **个人开发者的信息整理工具**：独立开发者或研究员可使用 NovaLink 整理个人关注的技术博客、在线工具与数据看板，配合标签与备注功能实现高效管理。
- **运维监控的辅助信息源**：运维工程师可将常用状态页面、云服务商健康状态检查链接纳入 NovaLink，配合可用性监控功能快速定位外部依赖故障。

## 快速开始

以下步骤适用于在 Linux 或 macOS 环境下快速启动 NovaLink 服务。

```bash
# 克隆项目仓库
git clone https://github.com/novalink-dev/novalink-station.git
cd novalink-station

# 安装依赖（使用 npm）
npm install

# 执行数据库初始化与种子数据加载
npm run init-db

# 启动开发服务器（默认监听端口 3000）
npm run dev
```

启动成功后，在浏览器中访问 `http://localhost:3000` 即可查看导航界面。生产环境部署建议使用 `npm run build` 构建静态文件，并通过 Nginx 或 Caddy 进行反向代理。

## 贡献指南

我们欢迎社区贡献者参与 NovaLink 的功能改进、文档完善与链接资源更新。请遵循以下流程：

1. **问题跟踪**：在 GitHub Issues 中查找 `help-wanted` 或 `good-first-issue` 标签的任务，或提交新 Issue 描述你发现的缺陷或期望的新特性。提交前请搜索是否已有类似议题。
2. **分支开发**：从 `main` 分支创建新的功能分支，分支命名遵循 `feature/描述` 或 `fix/描述` 格式。确保本地通过所有单元测试与 ESLint 检查。
3. **提交规范**：提交信息使用约定式提交格式（如 `feat: 增加按域名筛选功能` 或 `docs: 更新资源列表分类说明`）。每个提交应保持原子性，避免混合无关改动。
4. **测试验证**：对于新增的外部链接资源，请在 `seed.js` 中添加并同时提供简短的可用性验证记录。对于核心功能改动，需补充对应的单元测试用例。
5. **发起拉取请求**：将分支推送至 GitHub 仓库后，创建 Pull Request 至 `main` 分支。描述中需明确关联相关 Issue 编号，并附上手动测试截图或日志片段以供审阅。

## 常见问题

**问：为什么监控任务偶尔会报告某些链接不可用，但浏览器中能够正常访问？**

答：这通常是由于监控任务所在的网络环境（如云服务器）与本地客户端存在不同的 DNS 解析结果或防火墙策略。此外，部分站点可能会对频繁的 HEAD 请求进行限流。建议在配置文件中调整 `checker.userAgent` 字段模拟真实浏览器请求，并将 `checker.timeout` 适当增加至 10000 毫秒。如果问题持续，可将该节点加入 `monitor.ignoreList` 跳过主动检查，仅保留人工维护。

**问：如何迁移数据库到另一台服务器？**

答：NovaLink 使用 SQLite 作为数据存储，迁移时只需将项目根目录下的 `data/novalink.db` 文件复制到目标服务器的相同相对路径下，并确保目标服务器的 Node.js 版本与 SQLite 驱动兼容。若需跨操作系统迁移（如从 Windows 到 Linux），请使用 `.sql` 导出导入方式，避免字节序差异导致的兼容性问题。具体命令参考 `docs/deployment.md` 中的备份章节。

**问：能否在导航页面中嵌入自定义的 HTML 片段或 iFrame？**

答：出于安全考量，NovaLink 默认对用户添加的链接描述与备注字段进行了严格的 XSS 过滤，不支持直接嵌入 HTML 或 JavaScript。若需要在页面中嵌入外部仪表板，建议使用项目提供的 `custom-panel` 扩展点，通过后端代理方式将外部内容以安全沙盒形式加载。该功能目前处于实验阶段，详细配置方法请参阅 `docs/frontend-custom.md` 中的高级定制部分。
