# TechLink Navigator

TechLink Navigator 是一个面向开发者、技术研究人员与开源项目维护者的高质量技术资源外链汇总与导航系统。项目定位于解决技术信息过载背景下的资源筛选效率问题，通过人工筛选与社区投票机制，将分散于中文互联网环境中的优质技术文档、自建服务镜像、开源镜像站与社区论坛进行结构化整理与分类呈现。目标用户包括运维工程师、全栈开发人员、学术研究者以及独立开发者，帮助其在技术选型、环境配置、问题排查等场景下快速定位可靠的信息来源与工具入口。

## 功能概览

- **分类资源索引**：按技术领域、资源类型、维护状态等多维度对收录链接进行标签化管理，支持快速过滤与排序。
- **社区热度排序**：基于用户点击、收藏与评分数值动态调整资源展示权重，确保高价值条目保持曝光优先级。
- **可用性健康检查**：定时对收录的外链进行 HTTP/HTTPS 可达性探测，自动标记异常链接并通知维护者。
- **自定义收藏夹**：注册用户可创建个人资源收藏列表，支持添加私有备注与标签，便于团队内部共享知识库。
- **RSS 订阅源生成**：为每个分类目录生成标准 RSS 2.0 订阅地址，方便用户通过阅读器跟踪新增资源。
- **暗色主题与阅读模式**：内置明暗两套视觉方案，并提供去噪阅读模式，优化文档类链接的浏览体验。
- **外链跳转审计日志**：记录每次外链点击的来源 IP、时间戳与目标资源 ID，为反爬与异常行为分析提供数据基础。
- **开放 API 接口**：提供基于 RESTful 风格的数据查询接口，支持第三方工具批量获取资源列表与详情。

## 应用场景

- **开发环境快速初始化**：新员工入职或临时搭建开发环境时，可通过本导航站一键获取各镜像站地址、依赖包仓库与工具下载页，避免在搜索引擎中反复试错。
- **技术选型调研**：架构师在进行中间件选型或框架对比时，可利用本站的社区评分与评论聚合功能，快速筛选出口碑稳定且维护活跃的开源项目及其官方文档镜像。
- **离线文档与资源储备**：网络受限环境下的运维人员可定期通过本站的批量导出功能，获取常用资源列表及备用下载地址，用于内部镜像源的同步策略制定。
- **技术社区内容聚合**：技术博主或社区运营者可将本导航站作为内容素材池，快速发现近期讨论热度较高的新项目或工具版本更新动态。
- **学术研究数据采集**：高校研究人员在进行开源生态或中文技术社区演化分析时，可通过本站提供的分类数据接口获取结构化样本集，用于社会网络分析或主题建模。

## 快速开始

以下步骤将帮助您在本地环境中快速启动 TechLink Navigator 开发实例。

```bash
# 1. 克隆项目仓库
git clone https://github.com/techlink-navigator/navigator-core.git
cd navigator-core

# 2. 安装项目依赖（使用 npm）
npm install

# 3. 配置环境变量
cp .env.example .env
# 编辑 .env 文件，至少设置 DATABASE_URL 与 SESSION_SECRET

# 4. 初始化数据库结构
npm run db:migrate

# 5. 导入初始资源种子数据
npm run seed:resources

# 6. 启动开发服务器
npm run dev
```

访问本地服务 <code>http://localhost:3000</code> 即可查看导航站首页。生产环境部署请参考 `deploy` 目录下的容器化配置文件。

## 贡献指南

我们欢迎并感谢任何形式的贡献，包括但不限于新增资源链接、修复文档错误、改进代码性能或提交安全漏洞报告。请遵循以下步骤：

1.  **问题跟踪**：在提交代码或资源变更前，请先查阅 GitHub Issues 列表，确认是否已有相关讨论。若无，请新建 Issue 描述您的意图或发现的问题，获取社区反馈后再进行后续操作。
2.  **派生仓库**：将本仓库 Fork 至您的个人账户，并在本地创建功能分支（命名建议：`feature/资源分类-资源名称` 或 `fix/问题简述`）。确保分支基于最新的 `main` 分支。
3.  **提交规范**：代码提交时请使用语义化的 Commit Message（如 `feat: 新增容器化部署文档`、`fix: 修复健康检查超时未重置状态`）。资源新增请求需同时更新 `seed` 数据文件并附上资源简介与分类标签。
4.  **测试验证**：对于代码层面的改动，请确保新增或修改的功能已包含对应的单元测试或集成测试，且所有现有测试用例均能通过。运行 `npm run test` 进行本地验证。
5.  **发起拉取请求**：将您的本地分支推送至远程仓库后，向本仓库的 `main` 分支发起 Pull Request。请在 PR 描述中清晰关联相关 Issue 编号，并简述改动内容与测试覆盖情况。维护者将在 3 个工作日内进行 Review。

## 常见问题

**问：本站收录的资源链接是否保证永久可用？**

答：本站对所有外链提供被动健康检查服务，每 6 小时自动探测一次可用性，并在管理后台标记异常状态。但由于外部站点可能随时变更域名、停止服务或添加访问限制，本站无法保证任何第三方链接的永久可用性。我们建议用户在发现失效链接时，通过 Issue 或邮件渠道反馈，以便维护团队及时更新或移除。

**问：如何申请将自己的技术博客或开源项目加入导航站？**

答：请前往本站 GitHub 仓库，按照 `CONTRIBUTING.md` 中“资源新增请求”章节的模板提交 Issue，内容需包含资源名称、域名地址、所属分类（如“前端框架”“DevOps 工具”）以及 50 字以内的资源简介。维护团队将在收到请求后的 5 个工作日内完成人工审核，审核通过后随下次数据同步加入索引。请注意，本站暂不收录纯商业推广或涉及版权争议的链接。

**问：本地开发时如何模拟生产环境的健康检查与排名计算？**

答：您可以在 `.env` 文件中将 `CHECK_INTERVAL_MS` 调整为 60000（即 1 分钟），以便快速观察探测逻辑。同时，`ranking` 服务提供了 `--force-recalc` 命令行参数，运行 `npm run rank -- --force-recalc` 可手动触发全量排名重算，适用于本地调试算法参数。生产环境建议保持默认的 6 小时间隔，避免对目标站点造成不必要的请求压力。
