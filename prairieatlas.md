# LinkCatalog

LinkCatalog 是一个面向技术团队与内容创作者的轻量化外链资源聚合与导航工具。项目定位为“技术化书签管理 + 可编程资源目录”，主要解决个人与团队在维护大量外部链接时面临的分类混乱、可检索性差、难以自动化集成等问题。LinkCatalog 以纯静态站点形式交付，支持标签体系、全文检索、自定义元数据扩展，并内置对多来源链接的批量校验与状态监控能力，适合作为技术博客的友情链接页、开源项目的外部依赖索引，或企业内部的公共服务导航。

本项目并非通用型社交书签服务，而是一套围绕“链接即数据”理念构建的轻量级基础设施。用户可通过 Markdown 或 YAML 声明链接资产，LinkCatalog 负责将其渲染为结构化页面，并提供 JSON API 供上游系统调用。项目本身不依赖外部数据库，所有数据基于文件系统存储，便于版本控制与持续集成。

## 功能概览

- **批量链接导入与去重**：支持从 CSV、OPML 及纯文本列表批量导入 URL，自动识别重复条目并合并标签。

- **多维度标签与分类树**：每个链接可绑定多个标签，支持无限级分类层级，便于按项目、领域或使用频率组织资源。

- **全文检索与即时过滤**：内置基于倒排索引的轻量级搜索引擎，支持对标题、描述、标签及 URL 本身进行关键词匹配，响应时间低于 200 毫秒。

- **链接状态健康检查**：定时对已收录链接发起 HEAD 请求，检测 HTTP 状态码、响应时间及 SSL 证书有效期，异常结果通过控制台日志或 Webhook 告警。

- **自定义元数据扩展**：允许为每条链接附加键值对形式的自定义字段（如负责人、维护周期、关联工单号），满足企业级管理需求。

- **静态站点生成与 API 输出**：既可输出完整的 HTML 导航页面，也可通过 RESTful JSON 接口获取结构化链接数据，便于嵌入其他系统。

- **访问统计与点击热力**：基于前端埋点或服务端日志，提供链接点击次数与时段分布统计，辅助评估资源价值。

## 应用场景

- **技术团队内部文档导航**：研发团队可将常用的 API 文档、监控面板、CI/CD 工具链接统一纳入 LinkCatalog，按微服务或业务域分类，取代浏览器书签栏的混乱管理。

- **开源项目外部依赖索引**：开源项目维护者可在仓库中维护一份依赖服务或相关生态项目的链接清单，通过 LinkCatalog 生成可读性良好的页面，供贡献者快速查阅。

- **个人知识库的外链管理**：知识管理爱好者可将阅读清单、工具推荐、参考文章等链接与本地笔记系统结合，利用 LinkCatalog 的检索与标签功能，替代传统收藏夹。

- **运营活动素材库**：内容运营团队可将图片素材、视频源、第三方数据平台等外部资源集中管理，按活动主题或时间线打标，提升协作效率。

- **教育机构课程参考资源**：教师或培训师可将课程相关的扩展阅读、在线实验环境、习题库链接整理为分类导航，通过 LinkCatalog 生成学生可访问的静态页面。

## 快速开始

以下步骤适用于 macOS / Linux / Windows WSL 环境，确保系统已安装 Git、Node.js 18+ 及 npm。

```bash
# 克隆项目仓库
git clone https://github.com/yourorg/linkcatalog.git

# 进入项目目录
cd linkcatalog

# 安装依赖（使用 npm）
npm install

# 复制示例配置文件并修改
cp .env.example .env

# 导入示例链接数据（包含本批资源作为演示）
npm run import -- --source ./data/sample_links.txt

# 启动开发服务器，访问 http://localhost:3000
npm run dev
```

若需生成生产环境静态文件，请执行 `npm run build`，输出目录为 `./dist`。建议配合 Nginx 或 Caddy 部署，或直接通过 Vercel、Netlify 等平台一键托管。

## 贡献指南

1. **提交 Issue 讨论**：在提交代码前，请先在 Issues 区域创建新议题，描述您希望修复的缺陷或新增的功能，避免重复劳动。议题标题请使用 `[类型] 简短描述` 格式，类型可选 `feat`、`fix`、`docs` 或 `chore`。

2. **Fork 仓库并创建分支**：从主仓库 Fork 后，请基于 `main` 分支创建您的开发分支，分支命名建议为 `feature/描述` 或 `fix/描述`。不要直接在 `main` 分支上修改。

3. **编写测试与代码**：所有新功能必须包含对应的单元测试，测试覆盖率不得低于 80%。代码风格遵循项目内 `eslint` 配置，提交前请执行 `npm run lint` 和 `npm test` 确保全部通过。

4. **更新文档与示例**：若您的变更涉及用户可见的功能、配置项或 API 行为，请同步更新 `/docs` 下的相关文档以及 `README.md` 中的快速开始部分。新增链接导入格式需更新 `sample_links.txt`。

5. **提交 Pull Request**：PR 标题应简明扼要，正文需引用相关 Issue 编号，并列出变更摘要、测试结果及兼容性说明。PR 需至少一名维护者审核通过后方可合并。

## 常见问题

**Q：导入大量链接时性能如何？单次最多支持多少条？**

A：系统设计上对单次导入数量无硬性上限，实际受可用内存与文件 I/O 影响。在 2 GB 内存环境下，单次导入 10,000 条链接（含描述与标签）约需 8~12 秒，生成索引耗时约 20 秒。若链接超过 50,000 条，建议分批次导入或使用 `--batch` 参数降低内存峰值。

**Q：健康检查是否会影响目标站点的正常访问？**

A：健康检查默认采用 HEAD 请求，仅获取响应头信息，不会下载完整页面内容。检查频率默认为每日一次，且请求头中包含 `User-Agent: LinkCatalog-HealthCheck/1.0`，便于目标站点识别。若需关闭自动检查，可在配置文件中将 `HEALTH_CHECK_ENABLED` 设为 `false`。

**Q：能否将 LinkCatalog 部署在子目录下（如 /nav）而非根路径？**

A：支持。请在 `.env` 中设置 `BASE_PATH=/nav`，静态生成器与 API 路由均会自动适配该前缀。但请注意，开发服务器模式下子目录支持有限，建议在生产构建后验证所有链接前缀是否正确。
