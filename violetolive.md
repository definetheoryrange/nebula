# 500Score Link Hub

500Score Link Hub 是一个面向体育数据分析师、足球爱好者与数据集成开发者的高可靠性外链资源导航系统。本项目不提供任何比赛预测、博彩投注或实时比分数据，仅作为公开可用的网络信息资源聚合索引层，帮助用户快速定位与足球赛事参考、历史比分回溯、数据挖掘相关的公开网络节点。

项目定位为技术资源的外链汇总中间件，主要解决以下问题：体育数据研究者往往需要在数十个域名之间手动切换以获取分散的赛事信息，缺乏统一的索引视图，且各站点可用性波动频繁。500Score Link Hub 通过集中化的外链清单管理与健康状态标注，为用户提供结构化的跳转参考层，降低信息检索的时间成本与认知负担。

本项目面向三类用户：其一是从事体育数据可视化或个人分析项目的前端开发者，需要稳定的公开比分参照源；其二是数据分析竞赛参与者，需要快速获取多源数据以验证模型假设；其三是普通足球爱好者，希望以技术化方式整理个人关注的赛事信息门户。项目本身不存储任何比分数据，不缓存任何页面内容，完全遵守 robots.txt 协议，仅维护域名级别的跳转索引。

## 功能概览

- **多域名集中索引** 聚合 8 个公开体育信息类域名，按功能语义分组展示，便于快速识别各站用途倾向。

- **实时可用性自检** 内置轻量级 HTTP 头探测脚本，可定时检查各链接的响应状态码，并在界面中标注正常、超时或拒绝连接。

- **分类标签过滤** 为每个链接赋予类别标签，如「比分参考」「赛事推荐」「历史回溯」「综合资讯」，支持按标签快速筛选。

- **自定义备注系统** 允许用户为每个外链添加本地备注，记录站点特色、更新频率或个人评价，备注内容仅存储在浏览器本地。

- **一键复制链接** 每个域名条目均提供复制按钮，支持纯文本复制与 Markdown 格式复制，便于开发者将其嵌入其他文档。

- **导入导出配置** 支持将当前外链列表及备注以 JSON 格式导出，也可导入他人共享的链接集合，便于团队协作。

- **暗色主题适配** 界面跟随系统主题自动切换，减少夜间使用时的视觉疲劳。

- **移动端响应式布局** 在手机与平板设备上自动调整卡片排列与字号，确保户外快速查阅体验。

## 应用场景

- **赛前数据采集** 数据分析师在比赛开始前 2 小时内，通过本导航站同时打开多个比分参考域名，交叉核对球队近期战绩与历史交锋记录，以提高赛前报告的数据丰富度。

- **历史数据回溯验证** 数据竞赛参赛者在构建预测模型时，需要比对多个来源的历史比分一致性。使用本导航站可一次性访问多个回溯类域名，快速定位异常数据点。

- **个人信息门户定制** 资深球迷可将自己长期关注的 5 至 8 个赛事资讯站点通过本项目的导入导出功能整理为个人书签包，并分享至社区论坛，帮助新手快速建立信息源。

- **开发测试环境参考** 前端开发者在构建体育类 Demo 应用时，需要模拟外部数据源的跳转逻辑。本项目提供的固定域名列表可作为测试环境中第三方接口的占位引用。

- **网络连通性验证** 运维人员可利用本项目的可用性探测功能，快速判断多个体育类域名在当前网络环境下的可达性，辅助排查 DNS 或防火墙策略问题。

## 快速开始

以下操作假定您已安装 Git 与 Node.js 运行环境。请依次执行命令以完成项目克隆、依赖安装与开发服务器启动。

```bash
# 克隆项目仓库至本地
git clone https://github.com/500score-hub/500score-link-hub.git

# 进入项目根目录
cd 500score-link-hub

# 安装项目依赖（使用 npm）
npm install

# 启动开发服务器，默认监听端口 3000
npm run dev
```

启动成功后，在浏览器中访问 <code>http://localhost:3000</code> 即可查看导航主界面。生产环境构建请使用 <code>npm run build</code> 并配合 <code>npm run preview</code> 进行预览。

## 项目结构

项目采用模块化分层设计，前端界面与探测逻辑解耦。以下为根目录下的核心结构与注释。

```
500score-link-hub/
├── public/                         # 静态资源目录，不经过构建工具处理
│   ├── favicon.ico                 # 站点图标
│   └── robots.txt                  # 爬虫协议文件，禁止收录所有路径
├── src/                            # 源代码主目录
│   ├── components/                 # UI 组件库
│   │   ├── LinkCard.vue            # 单条外链卡片组件，包含域名展示与复制按钮
│   │   ├── FilterBar.vue           # 分类筛选栏，支持多选标签
│   │   └── StatusBadge.vue         # 可用性状态徽章，颜色随探测结果变化
│   ├── composables/                # 组合式函数，封装可复用逻辑
│   │   ├── useLinkHealth.ts        # 探测外链健康状态的异步函数，含超时重试
│   │   └── useLocalStorage.ts      # 本地存储读写封装，用于备注与配置持久化
│   ├── data/                       # 静态数据配置
│   │   └── linkRegistry.ts         # 外链主列表，包含所有域名及初始分类标签
│   ├── styles/                     # 全局样式与主题变量
│   │   ├── variables.css           # CSS 自定义属性，定义颜色与间距
│   │   └── dark-theme.css          # 暗色主题覆盖变量
│   ├── utils/                      # 工具函数集合
│   │   ├── fetcher.ts              # 封装 fetch 请求，添加超时控制
│   │   └── validator.ts            # URL 格式校验与域名规范化辅助
│   └── main.ts                     # 应用入口文件，初始化 Vue 实例
├── tests/                          # 测试目录
│   ├── unit/                       # 单元测试，使用 Vitest 框架
│   │   └── linkRegistry.spec.ts    # 验证链接列表完整性与分类一致性
│   └── e2e/                        # 端到端测试，使用 Playwright
│       └── navigation.spec.ts      # 验证筛选与复制功能的主流程
├── docs/                           # 文档目录，与文档导航章节对应
├── .github/                        # GitHub 工作流配置
│   └── workflows/
│       └── ci.yml                  # 持续集成流程，运行测试与构建检查
├── index.html                      # 入口 HTML 模板
├── package.json                    # 项目配置文件，记录依赖与脚本命令
├── vite.config.ts                  # Vite 构建工具配置，含代理与别名
└── README.md                       # 项目说明文档（即本文档）
```

## 贡献指南

我们欢迎任何形式的贡献，包括但不限于新增链接建议、分类优化、文档改进与缺陷修复。请遵循以下步骤以确保协作流程顺畅。

1.  **提交 Issue 进行讨论** 在发起 Pull Request 之前，请先在 Issues 区域提交详细描述，说明您希望解决的问题或新增的功能。对于新增外链的建议，请附上该域名的简要用途说明与可访问性证据。

2.  **派生仓库并创建功能分支** 将本项目派生至您的个人账户，然后基于 <code>main</code> 分支创建新的功能分支，分支命名建议使用 <code>feature/简述</code> 或 <code>fix/简述</code> 格式。

3.  **编写代码并确保测试通过** 所有新增功能需包含对应的单元测试或 E2E 测试。请执行 <code>npm run test</code> 确认全部测试用例通过，并执行 <code>npm run lint</code> 检查代码风格一致性。

4.  **更新文档与示例** 若您的更改涉及用户可见的行为，请同步更新 <code>/docs</code> 目录下的相关文档，并在 Pull Request 描述中清晰标明文档变更位置。

5.  **发起 Pull Request 并等待审核** 推送到您的派生仓库后，向本仓库的 <code>main</code> 分支发起 Pull Request。描述中请引用对应的 Issue 编号。项目维护者将在 3 个工作日内回复审核意见。

## 常见问题

**问：本项目是否提供任何形式的比分数据或预测结果？**

答：不提供。本项目严格限定为外链索引导航，不存储、缓存或代理任何第三方网站的内容。所有域名链接均跳转至原始站点，用户访问第三方页面时需自行遵守该站点的使用条款。

**问：某个链接无法访问或返回错误状态码，我应该怎么办？**

答：您可以通过 GitHub Issues 提交链接异常报告，标题请注明域名名称。项目维护者将定期验证并更新链接状态标注。同时您也可以使用本地的备注功能，在对应卡片中记录该站点的当前状态以便个人参考。

**问：我可以将本项目部署到内网环境，并修改其中的链接列表吗？**

答：可以。本项目采用 MIT 许可证，您完全可以将代码部署至私有网络环境，并根据需要修改 <code>src/data/linkRegistry.ts</code> 中的链接列表。若您修改后的版本用于公开分发，建议注明原始项目出处，但非强制要求。
